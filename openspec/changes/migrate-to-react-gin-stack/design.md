## Context

cvwonder-studio is currently a Next.js 15 monolith: React pages + API routes in one process, Prisma for PostgreSQL, and a downloaded `cvwonder` binary invoked via subprocess. This architecture requires Node.js in production, couples frontend and backend lifecycles, and makes it difficult to harden for Kubernetes (binary download at boot, no clean process isolation).

The target stack: a Go (Gin) binary that serves both the `/api/v1/*` REST API and the compiled React SPA via `go:embed`. One port, one process, one container. The `cvwonder` binary is bundled in the Docker image and invoked through a structured adapter interface designed to be swapped for direct Go package calls when cvwonder exposes public packages.

The current database schema (PostgreSQL) is preserved with minor adjustments. All session/theme logic migrates 1:1 in behavior; only the implementation layer changes.

## Goals / Non-Goals

**Goals:**
- Replace Next.js with a Go (Gin) HTTP server as the sole production process
- Embed the compiled Vite+React SPA in the Go binary (`go:embed`)
- Establish `ports.Generator` interface with a `binary.go` adapter (future: `library.go`)
- Replace Prisma with sqlc + pgx/v5 for type-safe SQL
- Replace Monaco Editor with CodeMirror 6
- Replace shadcn/ui with Radix UI primitives + Radix Colors (blue/navy palette)
- Bundle default themes in the Docker image; support runtime theme installation on PVC
- Local dev via docker-compose (postgres + Go with air + Vite dev server)
- Kubernetes manifests with SecurityContext, PVC, NetworkPolicy

**Non-Goals:**
- PDF export (separate plan)
- Admin UI (separate plan)
- cvwonder library adapter (interface designed now, implemented when cvwonder exposes `pkg/`)
- Multi-replica horizontal scaling (documented risk, not solved here)
- Auth / user accounts

## Decisions

### D1 — Gin over chi

**Decision**: Use `gin-gonic/gin` as the HTTP router.

**Rationale**: User requirement. Gin has a rich middleware ecosystem, built-in recovery/logging, and widespread production use. The hexagonal adapter pattern (ports/adapters) means the router choice doesn't affect business logic.

**Alternative considered**: chi (used in studio-03) — lighter, closer to stdlib, but not chosen per requirement.

### D2 — Vite + React SPA (no Next.js)

**Decision**: Pure SPA built with Vite, embedded in Go binary via `go:embed dist`.

**Rationale**: Eliminates Node.js in production. A single `GET /*` catch-all in Gin serves `index.html` for all non-API routes; the React router handles client-side navigation. Vite dev server proxies `/api/*` to the Go backend during local development.

**Alternative considered**: Keep Next.js with API routes removed — still requires a Node server, adds complexity for embedding, and defeats the "Go as sole server" goal.

### D3 — Binary adapter (now), interface designed for library (later)

**Decision**: Define `ports.Generator` interface; implement `adapters/cvwonder/binary.go` (exec subprocess). A future `adapters/cvwonder/library.go` adapter will call cvwonder Go packages directly when cvwonder moves its `internal/` packages to `pkg/`.

```
ports.Generator
  ├── adapters/cvwonder/binary.go    ← now: exec cvwonder CLI
  └── adapters/cvwonder/library.go  ← future: import cvwonder/pkg/*
```

**Rationale**: Keeps cvwonder stable. The adapter boundary in hexagonal architecture makes the swap zero-impact on business logic. YAML is always written to a temp file (never passed as CLI arg) to prevent injection.

**Alternative considered**: Docker sandbox (studio-03 approach) — not compatible with K8s without DinD; also eliminates the future library path.

### D4 — sqlc + pgx/v5 over Prisma

**Decision**: Hand-written SQL in `db/queries/*.sql`, type-safe Go code generated by sqlc, executed via pgx/v5 driver.

**Rationale**: No runtime ORM magic. Generated code is auditable. pgx/v5 is the standard high-performance PostgreSQL driver for Go. Prisma requires Node.js in the build pipeline.

**Alternative considered**: GORM — too much magic, harder to audit queries, performance overhead.

### D5 — CodeMirror 6 over Monaco

**Decision**: `@uiw/react-codemirror` with `@codemirror/lang-yaml` and AJV client-side validation.

**Rationale**: Monaco is a heavy dependency (~2MB gzipped) designed for VS Code-like experiences. CodeMirror 6 is modular, lighter, and already used in studio-03. Client-side AJV validation against the bundled schema provides inline diagnostics.

**Alternative considered**: Monaco — richer UX but payload cost not justified; incompatible with go:embed size targets.

### D6 — Themes bundled in image + runtime PVC

**Decision**: Default themes are copied into the Docker image during build. At startup, the backend registers them in the DB (idempotent upsert). Additional themes can be installed at runtime onto `/data/themes/` (PVC-mounted volume).

```
DB Theme table
  ├── localPath: /app/themes/<slug>    ← builtin (isBuiltin=true)
  └── localPath: /data/themes/<slug>  ← runtime installed (isBuiltin=false)
```

**Rationale**: Deterministic builds, no network dependency at boot. Runtime extensibility preserved for power users and admins.

**Alternative considered**: All themes from GitHub at runtime — adds network dependency, slow boot, poor K8s readiness probe behavior.

### D7 — go:embed for SPA

**Decision**: Gin serves the SPA via `go:embed dist` in the `pkg/embed` package. A catch-all `NoRoute` handler returns `index.html` for any non-API, non-preview path.

```go
//go:embed dist
var StaticFiles embed.FS

// In main.go:
r.Use(static.Serve("/", StaticFiles))
r.NoRoute(func(c *gin.Context) {
    c.FileFromFS("dist/index.html", http.FS(StaticFiles))
})
```

**Rationale**: Single binary deployment. No separate static file server or CDN required. In development, Vite serves the frontend with HMR, proxying API calls to Go.

### D8 — Radix Colors (blue/navy scale)

**Decision**: Import `@radix-ui/colors` blue and slate scales. Define semantic CSS tokens in `globals.css` using Tailwind v4 `@theme {}` syntax. Preserve the current design's blue/navy palette.

```css
@import "@radix-ui/colors/blue.css";
@import "@radix-ui/colors/blue-dark.css";
@import "@radix-ui/colors/slate.css";
@import "@radix-ui/colors/slate-dark.css";

@theme {
  --color-text-primary:    var(--blue-12);
  --color-surface-default: var(--slate-1);
  --color-accent:          var(--blue-9);
  --color-border:          var(--slate-6);
}
```

**Alternative considered**: Keep shadcn HSL vars — loses the Radix Colors dark-mode auto-switching, less idiomatic with Radix primitives.

## Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| Horizontal scaling blocked by ReadWriteOnce PVC | Document as known limitation; provide RWX PVC or object storage migration path in a follow-up plan |
| cvwonder binary size increases Docker image | Multi-stage build; binary is ~20MB; acceptable for the deployment target |
| PDF generation (future) needs Chromium in K8s pod, conflicts with readOnlyRootFilesystem | PDF is out of scope; when addressed, mount tmpfs on `/tmp` and use separate Pod SecurityPolicy |
| Big-bang migration — no incremental rollout | Run studio-03 (or current) in parallel until new stack is validated; use feature flag via env var if needed |
| CodeMirror loses Monaco's schema autocomplete richness | AJV inline validation + YAML language server (via `@codemirror/lang-yaml`) covers essential use cases; can add LSP later |
| sqlc codegen step in CI | Require `sqlc generate` in Makefile; add to CI pipeline; check-in generated files |

## Migration Plan

1. **Scaffold new repo structure** — `backend/`, `frontend/`, `infra/` directories
2. **Backend Phase 1** — domain models, ports, sqlc schema (mirrors Prisma schema), Gin server skeleton with health endpoints
3. **Backend Phase 2** — session usecases, theme usecases, cvwonder binary adapter, preview/validation routes
4. **Frontend Phase 1** — Vite setup, Radix Colors, Radix UI primitives, Zustand store
5. **Frontend Phase 2** — Landing page, Studio page (CodeMirror + preview iframe + theme selector)
6. **Integration** — go:embed dist, docker-compose local dev validated end-to-end
7. **Docker + K8s** — multi-stage Dockerfile, K8s manifests, SecurityContext, PVC
8. **Validation** — run against existing PostgreSQL DB from current studio; verify sessions, themes, generation

**Rollback**: Current Next.js studio remains deployable independently throughout; DNS cutover is the only shared concern.

## Open Questions

- Is `ReadWriteMany` PVC available in the target K8s cluster, or should we plan for a single-replica constraint from the start?
- Should the `cvwonder` binary version be pinned in the Dockerfile, or pulled from a release tag at build time?
- Should session token format change (current: UUID; studio-03: UUIDv7) or preserve UUID for DB compatibility?
